
下面以
```bash
python main.py -c configs/defaults.yaml
```
为例,只说明数据加载管线、数据组织、数据分布和评估协议这类方法无关内容.

## 1. 配置决定评估集

`main.py` 只解析一个参数: `-c/--config_file`. 配置文件读入后,所有字段都会挂到 `args` 上.

`configs/defaults.yaml` 中和评估集直接相关的字段是:

```yaml
path_dataset: ./data/
split: val
task_type: original
start_idx: 0
step_idx: 3
end_idx: 108
max_start_distance: hard
threshold_goal_distance: 1.0
max_steps: 250
```

这些字段组合后,默认命令会读取:

```text
scene root:   ./data/hm3d_v0.2/val
episode root: ./data/hm3d_iin_val
episode slice: sorted(episode_root/*)[0:108:3]
```

因此默认配置不是跑完整 108 个 episode,而是在排序后的 108 个 episode 中每 3 个取 1 个,共 36 个. 代码位置:

- `main.py`: 拼 `path_scenes_root_hm3d`、`path_episode_root`,然后切片 `start_idx:end_idx:step_idx`.
- `libs/experiments/task_setup.py::load_run_list`: 决定 episode 列表来自 episode root 还是已有结果里的 winners/failures/custom 列表.

如果 `run_list == ""`,直接读取 `episode_root` 下全部 episode 目录. 如果 `run_list` 是 `winners`、`failures`、`no_good` 或 `custom`,则从 `path_run/summary/<run_list>.csv` 读取 episode 名称,用于复跑已有结果的子集.

## 2. 数据组织形式

本仓库评估需要两类数据:

```text
data/
  hm3d_v0.2/
    val/
      <HM3D scene folders with *basis.glb>
  hm3d_iin_val/
    <episode_name>/
      agent_states.npy
      images/
        00000.png
        00001.png
        ...
      images_sem/
        00000.npy
        00001.npy
        ...
      goalImg.png
      goalImg_semantic.png
      obs_g.npy
      seen_but_unvisited_object_v2.npy
      nodes_gt_topological.pickle
      nodes_gt_topometric.pickle
      nodes_fast_sam_lightglue.pickle
      nodes_sam21_lightglue_depth_inferred.pickle
      reverse_goal.npy
```

`hm3d_v0.2/val` 是 Habitat/HM3D 场景资源,用于启动模拟器. `hm3d_iin_val` 是 ObjectReact/HM3D-IIN 评估 episode 数据,每个目录是一条任务.

episode 名称形如:

```text
4ok3usBNeis_0000000_chair_8_
```

含义大致是:

```text
<scene_id>_<episode_or_path_id>_<goal_category>_<goal_instance_id>_
```

运行时,`main.py` 会用 episode 名称第一个字段 `<scene_id>` 去 `./data/hm3d_v0.2/val` 下找对应 HM3D 场景目录,再找 `*basis.glb` 作为 Habitat scene.

默认 `goal_source: gt_metric` 时,评估主循环最关键的数据是:

- `agent_states.npy`: teach run/参考轨迹的 Habitat `AgentState` 序列.
- `images/`: teach run 上每个状态渲染出的 RGB 图像,主要供拓扑地图和可视化使用.
- `images_sem/`: teach run 上每个状态的语义 instance id 图.
- 在线仿真观测: 每一步由 Habitat 当前 agent pose 渲染 `color_sensor`、`depth_sensor`、`semantic_sensor`.

注意: `metadata.txt` 初始化时会写一次 `goal position`,但此时目标位置尚未完全 ready,所以默认结果里可能为空. 真正用于评估的是 episode 结束时追加的 `final_distance` 和 `distance_to_final_goal_from_start`.

## 3. 数据加载管线

一次默认评估的管线可以按下面理解:

```text
parse config
  -> build scene root and episode root
  -> init result directory and save args.yaml
  -> preload shared models
  -> list and slice episodes
  -> for each episode:
       find matching HM3D scene
       Episode(...)
         -> load agent_states.npy
         -> start Habitat simulator from *basis.glb
         -> compute semantic filters
         -> determine final goal
         -> select start state
         -> load optional map graph if goal_source needs it
         -> init goal generator/controller
       for step in max_steps:
         -> check success by geodesic distance
         -> render current observation from simulator
         -> split RGB/depth/semantic
         -> optionally infer depth/traversable mask
         -> compute method-specific goal input
         -> compute method-specific control/action
         -> execute action in simulator
         -> log pose, distance, action, collision
       -> close episode, write metadata/results
  -> write results_summary.csv
```

方法无关的关键点:

- episode 列表和起终点由配置、episode 数据、Habitat navmesh 决定.
- 控制器只负责每一步输出动作或速度.
- 成功/失败判断不依赖具体控制器,统一用当前 agent 到最终目标位置的 Habitat geodesic distance.

## 4. 目标和起点如何确定

`Episode.ready_agent()` 负责把一个 episode 变成一个可评估任务.

默认 `task_type: original` 且 `reverse: False`:

- `final_goal_position = agent_states[-1].position`
- `final_goal_image_idx = len(agent_states) - 1`

也就是说目标是 teach run 的最后一个 pose.

如果是 `alt_goal`,目标来自 `seen_but_unvisited_object_v2.npy`,代码读取其中的:

- `image_id`
- `instance_id`

然后在 HM3D semantic scene 中找到该 instance 的 3D 中心,把高度对齐到 teach run 平均地面高度,再 snap 到 navmesh.

如果是 `reverse`,目标不是原始终点,而是从原始起点沿 teach run 找到约 1m 外的状态,并反向设置朝向.

起点由 `max_start_distance` 控制:

```text
easy: 找到离目标 geodesic distance 约 3m 的 teach-run state
hard: 找到离目标 geodesic distance 约 5m 的 teach-run state
full: 使用 teach run 第一个 state
```

如果 `task_type == via_alt_goal`,hard 会额外加 3m 偏移. 如果 `reverse == True`,也会做 1m 偏移并反向设置朝向.

默认配置是 `hard`,所以每个 episode 会从 teach run 中选择一个离最终目标约 5m 的状态作为起点. 实际距离会因为 navmesh 和离散状态采样略有浮动.

## 6. 评估方法

在线运行时,每一步循环开头都会调用 `Episode.is_done()`:

```python
distance_to_goal = shortest_path(current_agent_position, final_goal_position)
success = distance_to_goal <= threshold_goal_distance
```

其中 shortest path 是 Habitat navmesh 上的 geodesic distance,不是欧氏距离. 默认阈值:

```text
threshold_goal_distance = 1.0m
```

所以只要 agent 当前位置到最终目标位置的 geodesic distance 小于等于 1m,该 episode 立即标记为:

```text
success_status=success
```

如果跑满 `max_steps` 仍未达到阈值,默认状态保持:

```text
success_status=exceeded_steps
```

如果 episode 初始化或运行中抛异常,`main.py` 会把异常类型、信息、文件名和行号写成 `success_status`,除非 `except_exit: True`.

主循环结束后,`utils.create_results_summary()` 汇总:

```text
Total Episodes
Successful Episodes
Failed Episodes
Success Rate (%)
Failure Reasons
```

这里有一个实现细节: `main.py` 中判断成功使用了

```python
success_status is None or "success" in success_status.lower()
```

正常状态 `success` 没问题;但如果某个异常字符串里意外包含 `success`,也会被计为成功. 后处理脚本 `scripts/evaluate_objecreact.py` 更严格,只把 `success_status == "success"` 算成功.

## 8. 后处理指标

`scripts/evaluate_objecreact.py <results-root>` 会递归查找时间戳结果目录,读取每个 episode 的 `metadata.txt` 和 `results.csv`,并计算:

- success rate
- SPL
- Soft SPL
- exceeded_steps / errors / no_status
- missing episodes
- ignored episodes

其中:

```text
shortest_path_length = distance_to_final_goal_from_start
remain_distance = final_distance
soft_spl = max(0, 1 - remain_distance / shortest_path_length)
```

如果成功,soft SPL 直接记为 1.0.

成功 episode 的 SPL 近似使用 `results.csv` 中 x-z 轨迹长度计算:

```text
path_length = sum(||xz[t+1] - xz[t]||) + remain_distance
spl = shortest_path_length / max(shortest_path_length, path_length)
```

脚本最终会用 denominator `len(episode_dirs) - num_ignored`,其中 ignored 来自 `configs/defaults.yaml` 的 blacklist.

因此要区分两层结果:

- `main.py` 在线运行的 `results_summary.csv`: 不应用 blacklist,只统计本次真正跑过的 episode.
- `scripts/evaluate_objecreact.py` 后处理结果: 会应用 blacklist,并额外计算 SPL/Soft SPL.
