**1. 为什么三个控制器的 `step_idx` 不同？它只是影响样本数吗？**

先说结论：`step_idx` 不是控制器行为参数，它是“抽哪些 episode 来跑”的参数。代码里就是直接对 episode 列表做切片：

[main.py](/home/xzy/projects/my_research/object-rel-nav/main.py:77)

```python
episodes = task_setup.load_run_list(args, path_episode_root)[
    args.start_idx : args.end_idx : args.step_idx
]
```

所以：

- `start_idx / end_idx / step_idx` 一起决定测试集子集
- `step_idx: 3` 表示每隔 3 个 episode 取一个
- `step_idx: 1` 表示全取

在你这个仓库默认值里：

- [configs/defaults.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/defaults.yaml:8) 是 `step_idx: 3`
- [configs/object_react.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/object_react.yaml:8) 是 `step_idx: 3`
- [configs/tango.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/tango.yaml:8) 是 `step_idx: 1`

所以默认确实不是同一个测试集规模。

但它**不只是样本数大小**，还影响“具体抽到哪些 episode”。因为这里是对排序后的 episode 列表做步长采样，[load_run_list()` 返回的也是排序结果](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:1047)。如果 episode 顺序本身带有场景/目标类别分布，`step_idx=3` 和 `step_idx=1` 不只是 36 vs 108，而是**评测子集组成也不同**。所以你说“不能直接横比”是对的。

**如果要做公平对比，至少要统一：**

- `start_idx`
- `end_idx`
- `step_idx`
- `task_type`
- `max_start_distance`

通常还建议统一：

- `split`
- `threshold_goal_distance`
- `max_steps`
- `reverse`
- `run_list / path_run`

**2. 哪些配置差异会影响评估结果？**

先说“直接定义评估协议”的几项，这些最该统一：

1. `start_idx / end_idx / step_idx`  
   决定评测 episode 子集。[main.py](/home/xzy/projects/my_research/object-rel-nav/main.py:77)

2. `task_type`  
   改任务定义：`original / alt_goal / via_alt_goal`。  
   它不仅换目标定义，还会影响 episode/map 路径选择，比如 `via_alt_goal` 会走不同 map 目录。[main.py](/home/xzy/projects/my_research/object-rel-nav/main.py:46)

3. `max_start_distance`  
   改起点离目标多远。代码里：
   - `easy` 约 3m
   - `hard` 约 5m
   - `full` 从轨迹起点开始  
   见 [select_starting_state()](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:1177)

4. `threshold_goal_distance`  
   这是成功阈值，直接决定 success/fail：  
   [is_done()](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:827)

5. `max_steps`  
   超过步数默认是失败，因为 `success_status` 初始就是 `"exceeded_steps"`。  
   见 [Episode 初始化](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:88) 和 [main 循环](/home/xzy/projects/my_research/object-rel-nav/main.py:106)

6. `split`  
   `train` 还是 `val`，当然会影响结果。[configs/defaults.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/defaults.yaml:16)

7. `reverse`  
   这会变成 reverse 任务，只支持 `task_type=original`。[main.py](/home/xzy/projects/my_research/object-rel-nav/main.py:222)

8. `run_list / path_run`  
   如果用了，不再是全量 episode，而是跑指定 winners/failures/custom 列表。[task_setup.py](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:1047)

然后说三份配置里，**确实会影响结果但更偏“方法设定”** 的差异：

- `method`
- `goal_source`
- `use_gt_localization`
- `infer_depth`
- `infer_traversable`
- `segmentor`
- `goal_gen.*`
- `cull_map_instances` 等

这些当然也会影响结果，但它们更像“方法的一部分”，不一定该统一。

你关心的三份配置的主要差异，可以粗分成这样：

**A. 评估协议差异**
- `step_idx`: `3 / 3 / 1`
- `max_steps`: `250 / 300 / 500`
- `use_gt_localization`: `True / True / False`
- `task_type`: 三者现在都是 `original`
- `threshold_goal_distance`: 三者现在都是 `1.0`
- `max_start_distance`: 三者现在都是 `"hard"`
- `split`: 三者现在都是 `"val"`

**B. 目标与地图定义差异**
- RoboHop: `goal_source: gt_metric`  
  [defaults.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/defaults.yaml:30)
- ObjectReact: `goal_source: gt_topometric`  
  [object_react.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/object_react.yaml:30)
- TANGO: `goal_source: topological`  
  [tango.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/tango.yaml:30)

这会直接改变 `Episode.get_goal()` 里生成 goal mask / planning signal 的方式。  
见 [task_setup.py](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:593)

**C. 感知设定差异**
- RoboHop: `infer_depth=True`, `infer_traversable=True`
- ObjectReact: `infer_depth=False`, `infer_traversable=False`
- TANGO: `infer_depth=True`, `infer_traversable=True`

这不是小差异，尤其对 TANGO 很重要，因为它会用 depth 和 traversable mask 走局部规划。[tango.py](/home/xzy/projects/my_research/object-rel-nav/libs/control/tango/tango.py:168)

**D. 拓扑/局部化参数差异**
ObjectReact 和 TANGO 的 `goal_gen` 差异很大，比如：

- `edge_weight_str`: `e3d` vs `margin`
- `loc_radius`: `16` vs `4`
- `subsample_ref`: `2` vs `1`
- `use_goal_nbrs`: `False` vs `True`
- `plan_da_nbrs`: `False` vs `True`
- `do_track`: `True` vs `False`

这些都会改 topological / topometric goal 生成和局部化行为。[task_setup.py](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:423)

所以如果你问“哪些最影响横向评估”：

**最该统一的：**
- `start_idx`
- `end_idx`
- `step_idx`
- `task_type`
- `max_start_distance`
- `threshold_goal_distance`
- `max_steps`
- `split`
- `reverse`

**不建议为了‘公平’强行统一的：**
- `method`
- `goal_source`
- `goal_gen.*`
- `infer_depth / infer_traversable`
- `segmentor`
- `use_gt_localization` 这个比较微妙，但它现在确实会显著影响结果

其中 `use_gt_localization` 我会单独提醒一下：  
它在 topological goal 里会让局部化走 oracle 分支，[loc_topo.py](/home/xzy/projects/my_research/object-rel-nav/libs/localizer/loc_topo.py:52)。这已经不只是“小实现差异”，而是实验假设差异。默认 RoboHop/ObjectReact 是 `True`，TANGO 是 `False`，这也会让结果不完全同口径。

**3. 为什么还有 `configs/controller/object_react_controller.yaml`？为什么没有三个 controller 配置文件？**

这个文件不是“总实验配置”，而是**learned controller 自己的模型配置**。

主配置文件做的是整场实验的配置，像：

- 跑哪些 episode
- 用什么任务
- 成功阈值
- 用哪种 method / goal_source
- 是否推 depth
- 是否推 traversable

而 `configs/controller/object_react_controller.yaml` 只在 `method == learnt` 时被加载：

[model_loader.py](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/model_loader.py:22)

```python
if method == 'learnt':
    from libs.control.objectreact import ObjRelLearntController
    goal_controller = ObjRelLearntController(
        config_filepath, goal_source=goal_source)
```

这个 controller 配置里放的是模型内部参数，比如：

- `goal_type`
- `obs_type`
- `dims`
- `context_size`
- `len_traj_pred`
- `close_threshold`
- `model_type`

见 [configs/controller/object_react_controller.yaml](/home/xzy/projects/my_research/object-rel-nav/configs/controller/object_react_controller.yaml:4)

所以层级上是：

- `configs/defaults.yaml` / `object_react.yaml` / `tango.yaml`  
  = 实验级配置
- `configs/controller/object_react_controller.yaml`  
  = ObjectReact 这个 learned controller 的模型级配置

为什么只有一个 controller 配置文件？

因为：

- RoboHop 的控制器基本是规则式函数，没有单独模型配置文件  
  见 [robohop.py](/home/xzy/projects/my_research/object-rel-nav/libs/control/robohop.py:6)
- TANGO 的控制器参数大多直接写在代码或由主配置提供，不是单独再读一个 yaml  
  见 [set_controller()](/home/xzy/projects/my_research/object-rel-nav/libs/experiments/task_setup.py:533)
- 只有 ObjectReact 是 learned controller，需要单独的网络/输入编码配置，所以有这个附属 yaml
